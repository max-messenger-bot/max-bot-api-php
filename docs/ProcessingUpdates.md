# Обработка обновлений

## Терминология

**Обновление** — это объект класса `\MaxMessenger\Bot\Models\Responses\Update` или наследуемого от него класса.
Тип обновления содержится в поле `update_type`.

**Событие** — это объект класса, наследуемого от `\MaxMessenger\Bot\MaxBot\Events\BaseEvent`.

## Рекомендации

- Используйте [рекомендации официального API](https://dev.max.ru/docs-api#Рекомендации%20по%20работе%20с%20API).
- Когда вы настраиваете получение **обновлений** о действиях в чат-боте, используйте:
    - **Long Polling** — для разработки и тестирования.
    - **Webhook** — для production-окружения.
- Для более продвинутого использования Вы можете изучить параметры и содержание методов из примеров и комбинировать
  их применение по своему усмотрению.
- Не сериализуйте и не сохраняйте объекты **событий**, вместо этого сохраняйте объекты **обновлений**.

## Способы получения обновлений

### Long Polling

Этот способ можно использовать для получения **обновлений** при разработке и тестировании,
если ваш бот не подписан на Webhook. Для production-окружения рекомендуем использовать **Webhook**.

Метод использует долгий опрос (long polling).
Каждое обновление имеет свой номер последовательности.
Свойство `marker` в ответе указывает на следующее ожидаемое обновление.

Все предыдущие обновления считаются завершёнными после прохождения параметра marker.
Если параметр marker не передан, бот получит все обновления, произошедшие после последнего подтверждения.

Обновления можно получить и обработать одним вызовом `MaxBot::handleFromServer()`, но Вы также можете изучить
код метода и изменить любую его внутреннюю часть, все используемые внутри методы являются публичными.

- Примеры [обработки обновлений](Examples/ProcessingUpdates.md)
- Больше информации в [официальной документации](https://dev.max.ru/docs-api/methods/GET/updates)

### Webhook

Способ доставки обновлений через **Webhook** — основной механизм получения событий в production-интеграциях.
При активной подписке **Long Polling** не работает.

Webhook-endpoint должен возвращать HTTP 200 в течение 30 секунд.
Любой другой код ответа или превышение тайм-аута — ошибка доставки.

Если доставка не удалась, выполняется до 10 повторных попыток с экспоненциально растущим интервалом:

- 1-я попытка: через 60 секунд
- 2-я попытка: через 150 секунд (60 × 2,5)
- 3-я попытка: через 375 секунд (150 × 2,5)
- и так далее

- Примеры [обработки обновлений](Examples/ProcessingUpdates.md)
- Больше информации в [официальной документации](https://dev.max.ru/docs-api/methods/POST/subscriptions)

#### Безопасность Webhook-запросов

Параметр **secret** позволяет убедиться, что Webhook-запросы приходят от MAX, а не от третьей стороны.

Если **secret** настроен, метод `MaxBot::makeUpdateFromGlobal()` автоматически проверит его в запросе.
Это необязательный параметр, но его настоятельно рекомендуется указывать.

## Обработка обновлений

Проще всего обрабатывать обновления с помощью бота реализованного в классе [MaxBot](../src/MaxBot.php).
Бот на основе **обновления** создаёт **событие** и даёт инструменты по обработке созданного **события**.

- Подробнее об обработке событий описано в разделе [Обработка событий](ProcessingEvents.md)
- Подробнее об обработке команд описано в разделе [Обработка команд](ProcessingCommands.md)

## Продвинутая обработка обновлений

> Этот раздел для опытных разработчиков

### Способы обработки

Вы можете сделать процесс обработки **обновлений** более прозрачным и добавить собственную функциональность
на любом этапе. Для этого существует ряд методов:

- Методы для **Webhook**:
    - **MaxBot::handleFromGlobal()** — Запускает процесс обработки **обновления** полученного из обрабатываемого
      запроса.
    - **MaxBot::makeUpdateFromGlobal()** — Создаёт объект **обновления** из обрабатываемого запроса.
    - **Update::newFromJsonString()** — Создаёт объект **обновления** из JSON-стоки.
- Методы для **Long Polling**:
    - **MaxBot::handleFromServer** — Получает **обновления** с сервера через API и запускает процесс их обработки.
      Ответ содержит маркер, который можно использовать в последующих запросах,
      чтобы предотвратить случайный пропуск обновлений из-за ошибок.
    - **MaxApiClient::getUpdates()** — Получает **обновления** с сервера и запускает процесс их обработки.
      Ответ содержит маркер, который можно использовать в последующих запросах,
      чтобы предотвратить случайный пропуск обновлений из-за ошибок.
- Общие методы:
    - **MaxBot::handleUpdate()** — Создаёт объект **события** на основе объекта **обновления** и
      запускает процесс обработки события. Ответ содержит `true`, если **событие** считается обработанным.
    - **MaxBot::makeEvent()** — Создаёт объект **события** на основе объекта **обновления**.
    - **MaxBot::handleEvent()** — Запускает процесс обработки **события** ранее добавленными обработчиками.
      Ответ содержит `true`, если **событие** обработано.
    - **MaxBot::handleEventUseHandlerList()** — Обрабатывает **событие** переданным списком обработчиков.
    - **BaseEvent::handle()** — Обрабатывает **событие* заданным обработчиком.

### Этапы обработки обновлений при использовании бота

- Сначала нужно получить **обновления** с сервера через API или из текущего запроса.
- **Обновление** может быть обработано немедленно или сохранено где-либо и обработано позже.
- Из **обновления** создаётся объект **событие**.
  Каждому типу **обновления** соответствует отдельный класс **события**.
  Объект **события** реализует вспомогательные функции для работы с обновлениями.
  Через объект **событие** можно получить объект **обновления**.
- **Событие** может быть обработано:
    - Обработчиками бота (методы `MaxBot::handleFromGlobal()`, `MaxBot::handleFromServer`, `MaxBot::handleUpdate()`,
      `MaxBot::handleEvent()`).
    - Своим списком обработчиков (метод `MaxBot::handleEventUseHandlerList()`).
    - Своим обработчиком (метод `BaseEvent::handle()`).
