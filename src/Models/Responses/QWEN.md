# Соглашения

## Модели ответов

- В качестве примера модели можешь брать модели Button, Chat, Attachment
- Модели ответов создаются используя статические методы самих моделей `newFromData()`, `newFromNullableData()`
- Списки моделей создаются используя статические методы самих моделей `newListFromData()`, `newListFromNullableData()`
- Двухуровневые списки моделей создаются используя статические методы `newList2DFromData()`, `newList2DFromNullableData()`
- Методы `newFromData()`, `newFromNullableData()`, `newListFromData()`, `newListFromNullableData()`, `newList2DFromData()`, `newList2DFromNullableData()` в модели добавлять не нужно, они должны наследоваться от родительской модели BaseResponseModel
- Класс модели должна иметь свойство $data с описанием свойств из схемы, при этом в описание не должны попадать унаследованные свойства
    - Свойства не присутствующие в required должны быть отмечены в описании как опциональные
    - Тип схемы array должен быть в описании phpdoc как тип list
    - Вложенные в array типы так же нужно документировать
    - Если свойство схемы имеет атрибут additionalProperties с указанием type и нет других вложенных свойств, используй типа формата `type[]`, где `type` - тип полученный из additionalProperties (именем свойства объекта могут быть строки и числа)
    - Вложенные объекты схемы нужно документировать как тип array, без подробностей
- Модель с discriminator в схеме, должны иметь метод `newFromData()` как в модели Attachment
    - Класс модели с discriminator не должен быть абстрактный, так как может создавать сам себя
    - В методе `newFromData()` значение для $className нужно получать точно так же, как это реализовано в модели Button
    - В методе `newFromData()` переменную $data нужно документировать не в описании метода, а перед первым использованием $data и документировать только те свойства, которые используются в методе
- Модель без discriminator в схеме, за исключением класса BaseResponseModel, не должны иметь метод `newFromData()`
- Проверка `if (static::class !== self::class)` в методе `newFromData()` нужна для исключения зацикливания при создании дочерних моделей
- Методы возвращающие bool свойства, должны иметь префикс is вместо get
- Для свойств схемы с временем, нужно добавлять метод с суффиксом `Raw` как это сделано в модели Chat
- Для свойств схемы содержащих перечисления, нужно добавлять метод с суффиксом `Raw` как это сделано в модели Button, при этом, не нужно использовать значение по умолчанию
- Для свойств схемы являющихся discriminator, нужно добавлять метод с суффиксом `Raw` как это сделано в модели Button
- Если свойство схемы имеет атрибут minLength больше 0, то в описании нужно использовать тип non-empty-string
- Если свойство схемы имеет атрибуты minLength, minItems больше 0, то в описании к методу с префиксом get в скобках нужно добавить описание как в методе `Button::getText()`
- Если свойство схемы имеет атрибуты maxLength, maxItems, то в описании к методу с префиксом get в скобках нужно добавить описание как в методе `Button::getText()`
