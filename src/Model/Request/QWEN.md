# Соглашения

## Модели запросов

- В качестве примера модели можешь брать модели `BotCommand`, `BotPatch`, `Button`, `PhotoToken`
- В конструкторе и методе `new` все аргументы должны иметь значение по умолчанию
- Конструктор устанавливает напрямую только свойства для чтения, для остальных обязательно использовать сеттер
- Должен быть реализован статический метод `new()` с такими же аргументами (и описаниями к ним) как у конструктора, внутри надо вызывать конструктор
- Должен быть реализован статический метод `make()` с такими же аргументами (и описаниями к ним) как у конструктора, но `required` свойства схемы должны быть обязательными аргументами
- Класс модели запроса должен иметь свойство $data с описанием свойств из схемы, при этом в описание не должны попадать унаследованные свойства
    - Свойства отсутствующие в `required` схемы, должны быть отмечены в описании как опциональные `type?`
    - Свойств `nullable` в схеме быть не должно, нужно удалить подтип `null` и удалить это свойство из `required` схемы
    - Свойства отмеченные как `deprecated`, нужно игнорировать, а так-же их не должно быть в описании $data
    - Тип схемы `array` должен быть в описании свойств $data как тип `list` или 'non-empty-list' (если есть `minItems`)
    - Если схема модели имеет атрибут `additionalProperties`, используй тип для свойства в $data тип `object`
    - Вложенные типы так же нужно документировать
    - Свойства `notify` и `quick` в $data всегда обязательные
- Для всех свойств схемы должен быть метод с префиксом `isset`
    - Свойства, которые устанавливаются в конструкторе и не имеют подтипа`null` и свойств только для чтения не должны иметь метод с префиксом `isset`
- Для всех свойств которые могут отсутствовать в $data (`type?`), должен быть метод с префиксом `unset`, у других свойств, метода с префиксом `unset` быть не должно
    - У следующих моделей не должно быть методов с префиксом `unset`: PhotoAttachmentRequestPayload
- Если свойство схемы имеет атрибут `minLength` больше 0, то в описании свойства в $data нужно использовать тип `non-empty-string`
- Если свойство схемы имеет атрибуты `minLength`, `minItems` больше 0, то в описании к аргументам конструктора, `new()`, `make()` и методам с префиксом `new`, `set` в скобках нужно добавить описание как в методе `Button::getText()`
- Если свойство схемы имеет атрибуты `maxLength`, `maxItems`, то в описании к аргументам конструктора, `new()`, `make()` и методам с префиксом `new`, `set` в скобках нужно добавить описание как в методе `Button::getText()`
- В конструкторе и сеттерах с аргументами из схемы, вместо типов `list` и `non-empty-list`, нужно использовать типы `array` и `non-empty-array`, а в коде при сохранении в $data, делать преобразование функцией `array_values()` (смотри пример в модели `BotPatch`)
    - К другим типам это правило не применять
- В методах с префиксом `get`, `isset`, `unset` генерировать описание не нужно
- Для методов с префиксами `set` и `unset` добавь в описание: `@return $this`
- В методах с префиксом `set` описание из API размести в описании параметра метода
- В методах с префиксом `set` аргументы должны быть обязательными
- В методах с префиксом `set` добавь валидаторы по атрибутам `minLength`, `maxLength`, `minItems`, `maxItems` и `pattern`
- Если функция возвращает `$this`, это обязательно нужно документировать: `@return $this`
- Свойство схемы содержащее `discriminator` должно быть только для чтения
- Валидацию входных данных в методе `new()` и `конструкторе` делать не нужно
- Поправки к схеме которые нужно обязательно учитывать
    - Свойства $data `token`, `code`, `message_id` не могут быть пустой строкой, добавляй в комментарий `(minLength: 1)`, используй вместо типа `string` тип `non-empty-string` и делай проверку методом `self::validateString()`
    - У модели `SubscriptionRequestBody` в схеме нет свойства `version`, но оно должно быть в модели
- Если методы ValidateTrait не используются, не добавляй его в класс
- Если в $data нет `required` свойств и в конструкторе не меняется `$this->requiredOnce`, не добавляй в класс ValidateRequiredTrait
- Если в $data модели есть обязательные свойства (не отмеченные `?`), нужно поместить их список в свойство `$this->required` и подключить трейт `ValidateRequiredTrait`
    - `$this->required` устанавливается до вызова parent метода.
    - В абстрактных классах нужно не присваивать `$this->required`, а обновлять (смотри пример в модели `Button`).
    - В этот список не нужно добавлять свойства только для чтения и те, которые устанавливаются в конструкторе и не имеют подтипа`null`
